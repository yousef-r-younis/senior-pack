The 5 Phases Every Project Needs

Your Project Lifecycle

Phase 1: Research & Planning (Weeks 1-2)

  • Literature review - what's been done before?
  • Technology stack decisions
  • Requirements gathering
  • Create initial system design
  • Budget and resources assessment

Our example: Researched OpenWrt capabilities, VPN protocols, Raspberry Pi specs

Phase 2: Design & Architecture (Weeks 3-4)

  • Detailed system architecture diagrams
  • Database schema (if applicable)
  • API design
  • User interface mockups
  • Define success metrics

Our example: Drew network topology, designed web interface wireframes

Phase 3: Implementation (Weeks 5-10)

  • Build MVP features first
  • Develop layer by layer
  • Version control EVERYTHING
  • Test as you go
  • Weekly progress reviews

Our example: Hardware ‚Üí Firmware ‚Üí Software ‚Üí Interface

Phase 4: Testing & Refinement (Weeks 11-12)

  • Integration testing
  • Bug fixes
  • Performance optimization
  • User testing (if applicable)
  • Add nice-to-have features

Phase 5: Documentation & Presentation (Weeks 13-14)

  • Write report (<100 pages!)
  • Create presentation slides
  • Prepare demo
  • Practice, practice, practice

Setting Clear Milestones

How to Prove Progress to Your Advisor

‚úÖ What Makes a Good Milestone?

Specific: "Basic routing works" not "Make progress on networking"
Testable: Can you demo it? Can you show it working?
Time-bound: "By Week 6" not "Eventually"
Visible: Produces tangible output (code, hardware, docs)

Our Router Milestones - Examples

‚úÖ Good Milestones

  • Week 5: Raspberry Pi boots OpenWrt and connects to internet
  • Week 7: VPN tunnel established and traffic routes through it
  • Week 9: Web interface displays connected devices
  • Week 11: AdGuard blocks ads on test device

‚ùå Bad Milestones

  • "Work on hardware stuff"
  • "Research networking"
  • "Make the website look nice"
  • "Fix bugs"

Too vague! Can't test or demo these.

üí° Pro Tip: During weekly advisor meetings, come prepared with what you accomplished, what's next, and where you're stuck. Advisors love seeing clear progress!

When to Pivot vs. Push Through

Knowing When to Adjust Your Plan

üîÑ Signs You Should PIVOT:

  • Technical impossibility: "This hardware literally can't do what we need"
  • Timeline disaster: "We're Week 10 and have 10% done"
  • Budget explosion: "We need $500 more we don't have"
  • Dependency failure: "The API we rely on shut down"
  • Scope creep gone wild: "We keep adding features and nothing's finished"

üí™ Signs You Should PUSH THROUGH:

  • Typical bugs: "It's not working yet but we know why"
  • Learning curve: "This is hard but we're making progress"
  • Minor delays: "We're 1 week behind but can catch up"
  • Advisor confidence: "Your advisor thinks you're on track"
  • 80% there: "Most features work, just polishing now"

Our Real Pivot Example:

Original plan: Build custom voice assistant from scratch using speech recognition libraries.

Problem (Week 9): Speech recognition accuracy was terrible, taking too much time to debug.

Pivot decision: Integrated existing voice command library instead of building from scratch.

Result: Got voice commands working in 2 days instead of 2 weeks. Still impressive, saved the timeline!

üí° Golden Rule: Pivot on HOW, not WHAT. Keep your core project vision, but be flexible on implementation details.

Working with a Project Partner

Collaboration Best Practices

‚úÖ How We (Yousef & Moayad) Divided Work:

Yousef's Focus:

  • Hardware setup & integration
  • OpenWrt firmware configuration
  • VPN implementation
  • Project documentation

Moayad's Focus:

  • Web interface development
  • API endpoints
  • Traffic monitoring features
  • Testing & debugging

Shared responsibilities: Design decisions, advisor meetings, presentation prep, report writing

Communication Strategy:

  • Daily standup (async): WhatsApp message each morning - what you did yesterday, what you're doing today, any blockers
  • Weekly sync meeting: 1-hour meeting to review progress, plan next week
  • Shared documents: Google Docs for report, GitHub for code, Trello for tasks
  • Clear ownership: Each person owns specific components - reduces conflicts

‚ö†Ô∏è Avoid Partner Pitfalls:

  • Unclear division of work: Both think the other is doing X ‚Üí nothing gets done
  • No communication: "I thought you were working on that!"
  • One person doing everything: Unfair and risky if they get sick/busy
  • Conflicting Git commits: Use branches, communicate before merging

Dealing with Setbacks (They WILL Happen)

Murphy's Law: What Can Go Wrong, Will Go Wrong

Our Real Setbacks & How We Handled Them:

Setback #1: Unexpected hardware issues and missing equipment (Week 5)

What happened: We were missing a WiFi USB dongle that we needed for the router setup. We couldn't find it anywhere online - it was out of stock everywhere.

The panic moment: We thought we'd have to change our entire project because we couldn't get the hardware we needed.

How we handled it:

  • Started asking around - talked to people we knew
  • An instructor had a matching WiFi USB dongle
  • They lent it to us, solving the problem immediately
  • We could move forward with our original design

Lesson: Your network is your lifeline! Build relationships with instructors and classmates. Someone often has what you need.

Setback #2: OpenWrt Update Broke Everything (Week 9)

What happened: Updated OpenWrt firmware, all our custom configs disappeared.

How we handled it:

  • Rolled back to previous firmware version
  • Documented our configs in Git (should've done earlier!)
  • Created installation script for reproducibility
  • Decided to freeze firmware version until after presentation

Lesson: Don't update critical dependencies late in development!

Setback #3: Midterm Week Conflict (Week 7)

What happened: Both of us had 6 midterms that week, zero project time.

How we handled it:

  • Planned for this in our timeline - had buffer
  • Communicated with advisor in advance
  • Doubled down the week after to catch up

Lesson: Account for university calendar when planning!

üí° General Setback Strategy:

  1. Don't panic: Take a breath, assess calmly
  2. Document what broke: Write it down
  3. Estimate impact: How many days/weeks will this cost?
  4. Communicate immediately: Tell your partner and advisor
  5. Find workaround: Can you substitute? Simplify? Delay a feature?
  6. Learn from it: How do you prevent this next time?

Key Takeaways

Planning & Execution Essentials üéØ

‚úÖ Remember These:

Break complex projects into independent layers (hardware ‚Üí firmware ‚Üí software ‚Üí interface)
Follow the 5 phases: Research ‚Üí Design ‚Üí Implementation ‚Üí Testing ‚Üí Documentation
Create realistic timelines with 20-30% buffer time
Set SPECIFIC, TESTABLE milestones you can demo to your advisor
Use Git from Day 1 - version control is non-negotiable
Know when to pivot (technical impossibility) vs. push through (typical bugs)
Communicate constantly with your partner - clear ownership prevents conflicts
Expect setbacks and plan for them - they're normal!

üí° The 80/20 Rule Recap:

Complete 80% of your project in the first 60% of your timeline. This gives you breathing room for:

  • Unexpected bugs and hardware failures
  • Midterm/exam weeks
  • Report writing and presentation prep
  • Adding bonus features that make it shine

üö® Don't Make These Mistakes:

  • Starting without a clear timeline
  • No version control until things break
  • Waiting until Week 12 to start documentation
  • Poor communication with partner/advisor
  • Adding new features when you should be testing
