Activity Diagram

Shows the flow of activities and decision points in a process or workflow.

Figure 1: Activity diagram showing process flow with decision points

üìå When to Use

  • Modeling business processes
  • Showing workflow logic
  • Documenting decision paths
  • Implementation chapter

üîë Key Elements

  • Start/End nodes (filled circles)
  • Actions (rounded rectangles)
  • Decision diamonds
  • Arrows showing flow direction

üí° Tip

Keep activity diagrams focused on one specific process. If a diagram gets too complex, split it into sub-processes.

Class Diagram

Shows the structure of your code ‚Äî classes, attributes, methods, and relationships.

Figure 2: Class diagram showing system code structure and relationships

üìå When to Use

  • Design & architecture chapter
  • Object-oriented systems
  • Showing code organization
  • Database-related structures

üîë Key Elements

  • Classes (name, attributes, methods)
  • Associations (lines connecting classes)
  • Inheritance (arrow with hollow head)
  • Multiplicity (1..*, 0..1, etc.)

üí° Tip

Don't include every single class ‚Äî focus on the key classes that illustrate your system's architecture. Evaluators want to see your design decisions, not every getter/setter.

Use Case Diagram

Shows how different users (actors) interact with your system's features.

Figure 3: Use case diagram showing actor-system interactions

üìå When to Use

  • Requirements chapter
  • Defining system scope
  • Communicating with stakeholders
  • Early design phase

üîë Key Elements

  • Actors (stick figures)
  • Use cases (ovals)
  • System boundary (rectangle)
  • Relationships (include, extend)

üí° Tip

Group related use cases together and clearly label each actor. This diagram is great for showing evaluators the full scope of your project at a glance.

Sequence Diagram

Shows how components interact with each other over time, step by step.

Figure 4: Sequence diagram showing component interactions over time

üìå When to Use

  • API call flows
  • Authentication processes
  • Client-server interactions
  • Implementation chapter

üîë Key Elements

  • Lifelines (vertical dashed lines)
  • Messages (horizontal arrows)
  • Activation bars (thin rectangles)
  • Return messages (dashed arrows)

üí° Tip

Number your messages sequentially and keep the diagram reading left-to-right for the main flow. Add notes for complex steps that need extra explanation.

System Architecture Diagram

Shows the overall structure and components of your system and how they connect.

Figure 5: System architecture diagram showing overall project structure

üìå When to Use

  • Design chapter (essential!)
  • Project overview
  • Showing technology stack
  • Presentations

üîë Key Elements

  • Components/modules (boxes)
  • Connections (arrows)
  • Data flow direction
  • External services/APIs

üí° Tip

This is usually the first diagram evaluators look at. Make it clean, well-labeled, and ensure it tells the story of your entire system in one view.
